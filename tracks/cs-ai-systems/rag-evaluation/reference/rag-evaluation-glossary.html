RAG Evaluation Glossary

A compact reference for the RAG evaluation module. Use this when labeling failures, choosing metrics, or designing the PMP-docs eval dataset.

The Three-Level Split

Retrieval evaluation

Checks whether the system found the right source chunks for the question. Failures usually involve chunking, embeddings, top-k, ranking, or missing indexed material.

Generation evaluation

Checks whether the answer used the retrieved context correctly. Failures usually involve hallucination, contradiction, incomplete synthesis, or prompt behavior.

End-to-end evaluation

Checks the whole path from user question to final answer. Useful for product quality, but less diagnostic unless retrieval and generation signals are logged separately.

The module's first habit: never call an answer "bad" until you can say which part of the RAG loop made it bad.

Metric Vocabulary

Metric Primary target Question it asks
Context precision Retriever ranking Are the most relevant chunks ranked above weaker chunks?
Context recall Retriever coverage Did the retrieved context include the facts needed for the expected answer?
Context relevancy Retriever noise Are the retrieved chunks actually relevant to the input?
Answer relevancy Generator usefulness Does the answer address the user question?
Faithfulness or groundedness Generator grounding Is the answer supported by the retrieved context?
Answer correctness End-to-end answer Does the answer match the reference answer or expected facts?

Primary sources: DeepEval RAG evaluation guide, Ragas metrics docs, and the RAGAS paper.

Failure Labels

Label Use when Next experiment
missing_source The corpus does not contain the needed fact. Add or update source material; do not tune retrieval first.
retrieval_miss The corpus has the fact, but top-k results omit it. Change chunking, query rewriting, embedding model, or top-k.
retrieval_noise The fact appears, but irrelevant chunks crowd the context. Adjust chunk size, filters, reranking, or top-k.
generation_drift The retrieved context is enough, but the answer adds or changes facts. Change prompt, model, grounding rule, or refusal behavior.
answer_incomplete The answer is grounded but misses required facts. Clarify expected facts and prompt for complete coverage.
dataset_ambiguous The expected answer or expected source is underspecified. Repair the eval case before judging the system.

Dataset Row Skeleton

Keep the first dataset boring and explicit. Each row should make diagnosis possible without guessing.

{
  "id": "pmp-001",
  "question": "What is the first 8-12 week milestone?",
  "expected_facts": ["Produce a portfolio-grade AI backend system"],
  "expected_sources": ["PROGRAM.md"],
  "retrieved_chunks": [],
  "actual_answer": "",
  "failure_labels": []
}

If the dataset row does not name expected facts and expected sources, most metric scores become harder to interpret.
